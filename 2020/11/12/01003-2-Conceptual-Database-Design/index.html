Conceptual Database Design

Introduction

Conceptual Database Design: A technique for understanding and capturing business information requirements graphically.

Entity Relationship Model

Entity and Relationship types, attributes

  • Entity: an individual object, e.g., a person, place, object, event, or concept about which you want to gather and store data.
  • Entity Type (also: entity set): a collection of entities that share common properties or characteristics
  • Attribute: describes one aspect of an entity set
  • Domain: possible values of an attribute
    • simple/composite/single-valued/multi-valued
  • Key: minimal set of attributes that uniquely identifies an entity in the set (several such Candidate Keys possible)
  • Entity Schema: entity set name, attributes (+domains), PK
  • Relationship: relates two or more entities
  • Relationship Type (Relationship Set): set of similar relationships
  • Relationship-Attribute: Relationships can also have additional properties
  • Relationship-Role: Each participating entity can be named with an explicit role
  • Relationship Type Schema: Relationship name, Role names (optional), Relationship attributes (+domains)

Key, participation and cardinality constraints

The combination of the primary keys of the participating entity sets forms a super key (superset of a key) of a relationship.

Key Constraints

  • N:1 relationship: at most one
    Representation: $\rightarrow $
  • 1:N (total participation): at least one
    Representation: thick line
  • 1:1 relationship: exactly one
    Representation: thick $\rightarrow $
  • N:N relationship: line

Cardinality Constraints

how often an entity of set E participates in R at least (minimum cardinality) and at most (maximum cardinality). ‘*’ as max

Weak entities, IsA hierarchies, aggregation

Weak entity type

  • An entity type that does not have a self-contained primary key, must:
    • relate to the identifying owner entity set via a one-to-many identifying relationship type from the identifying owner entity set to the weak entity set
    • have total participation in the identifying relationship type
  • The discriminator (or partial key) of a weak entity type is the set of attributes that
    distinguishes among all the entities of a weak entity type related to the same owning entity.
  • The primary key of a weak entity type = the primary key of the strong entity type(s) on which the weak entity type is existence dependent, plus the weak entity type􏰲􏰹 discriminator.

Enhanced E-R Model

E-R Model + SPECIALIZATION/ GENERALIZATION +ABSTRACTIONS 􏰼􏰷(aggregation􏰲)

Generalisation/ Specialisation/ Inheritance

  • Attribute inheritance 􏰵 a lower-level (subclass) entity type inherits all the attributes and
    relationship participations of its supertype.
Constraints on IsA Hierarchies:
  • Overlap Constraints
    • Disjoint: an entity can belong to only one lower-level entity set
    • Overlapping: an entity can belong to more than one lower-level entity set
  • Covering Constraints
    • Total: an entity must belong to one of the lower-level entity sets (thick line)
    • Partial (the default): an entity need not belong to one of the lower-level entity sets

Aggregation

We can convey relationships between an entity set and another relationship type via aggregation
